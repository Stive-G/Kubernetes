Application microservices

L'objectif est de passer d'un projet microservices local à une application conteneurisée, orchestrée par Kubernetes et accessible via un point d'entrée HTTP unique.

Contexte

Une application découpée en microservices

Chaque service a une responsabilité précise et communique avec les autres via un bus d'événements.

ServiceRôle
clientInterface React visible par l'utilisateur
postsCréation et gestion des sujets
commentsCréation et gestion des commentaires
queryAgrégation des posts et commentaires
moderationValidation ou rejet des commentaires
event-busTransmission des événements entre services

Le projet n'est pas monolithique : chaque service a une responsabilité précise et communique avec les autres via un bus d'événements.

Motivation

Pourquoi passer de Docker à Kubernetes ?

Docker crée les images ; Kubernetes les déploie, les supervise et assure leur disponibilité.

DockerKubernetes
Lance un conteneurOrchestre plusieurs conteneurs
Pas d'auto-réparationRecrée automatiquement les Pods
Pas de découverte de servicesServices Kubernetes avec DNS interne
Pas de routage HTTP natifIngress
Gestion manuelleOrchestration automatique

Docker permet de créer les images des conteneurs, tandis que Kubernetes permet de les déployer, de les superviser, de les faire communiquer et d'assurer leur disponibilité.

Build & vérification des images

Docker prépare les images ; Kubernetes les exécute ensuite sous forme de Pods.

docker build -t posts ./posts
docker build -t comments ./comments
docker build -t query ./query
docker build -t moderation ./moderation

docker build -t client:v2 ./client

# Vérification
docker images

imagePullPolicy: IfNotPresent — Kubernetes utilise les images locales lorsqu'elles sont disponibles, tout en permettant leur téléchargement si nécessaire.

  • Une image a été construite pour chaque microservice.
  • L'image réellement utilisée pour le client est client:v2.
  • Ces images sont référencées ensuite dans les Deployments Kubernetes.

Déploiement Kubernetes

Les Deployments

Le Deployment indique à Kubernetes quel conteneur lancer, combien d'instances maintenir et comment identifier les Pods associés.

Rôle d'un Deployment

  • créer les Pods ;
  • maintenir les Pods en fonctionnement ;
  • redémarrer automatiquement un Pod en cas de panne ;
  • gérer les mises à jour.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: posts-depl
spec:
  replicas: 1
  selector:
    matchLabels: { app: posts }
  template:
    metadata:
      labels: { app: posts }
    spec:
      containers:
        - name: posts
          image: posts
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 4000

Le Deployment indique à Kubernetes quel conteneur lancer, combien d'instances maintenir et comment identifier les Pods associés.

Déploiement Kubernetes

Services : adresse stable pour les Pods

Les Pods peuvent changer d'adresse IP. Le Service permet donc aux autres microservices de les joindre avec un nom stable.

apiVersion: v1
kind: Service
metadata:
  name: posts-clusterip-srv
spec:
  type: ClusterIP
  selector: { app: posts }
  ports:
    - name: posts
      protocol: TCP
      port: 4000
      targetPort: 4000
Service KubernetesPortCible
client-srv3000client
posts-clusterip-srv4000posts
comments-srv4001comments
query-srv4002query
moderation-srv4003moderation
event-bus-srv4005event-bus

Tous les Services sont de type ClusterIP, car ils servent principalement à la communication interne dans le cluster. L'accès externe est géré par l'Ingress.

Exemple — création d'un commentaire

  • 1. comments crée le commentaire
  • 2. comments émet CommentCreated
  • 3. event-bus transmet à moderation
  • 4. moderation valide ou rejette
  • 5. moderation émet CommentModerated
  • 6. comments met à jour le commentaire
  • 7. query met à jour l'affichage

Exposition

Configuration de l'Ingress

Le contrôleur Ingress NGINX reçoit le trafic HTTP local, puis applique les règles définies pour router les requêtes.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-srv
  annotations:
    nginx.ingress.kubernetes.io/use-regex: "true"
spec:
  ingressClassName: nginx
  rules:
    - http:
      paths:
        - path: /posts/create
          pathType: Prefix
          backend:
            service:
              name: posts-clusterip-srv
              port: { number: 4000 }

ingressClassName: nginx permet d'associer explicitement la ressource Ingress au contrôleur NGINX.

# Accès local via port-forward
kubectl port-forward -n ingress-nginx \
 service/ingress-nginx-controller 8080:80

# Application accessible sur
http://localhost:8080

Le contrôleur Ingress NGINX reçoit le trafic HTTP local, puis applique les règles définies dans l'Ingress pour router les requêtes vers les bons Services Kubernetes.
